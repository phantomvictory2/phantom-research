## 1. Executive Summary

**Verdict: CONDITIONAL GO.**

The pipeline is wired end-to-end and the data layer is sound: the collector is genuinely writing, the clean views exist and their contamination arithmetic reproduces independently, and the dashboard reads live values rather than hardcoded numbers. Two independent computations of contamination agree (17.4%), which is the strongest single piece of evidence that the cleaning layer is doing what it claims.

**But three findings block an unconditional GO:**

1. **The headline 66.3% / 90.4% "BTC↔Polymarket agreement" is not a research finding.** It is a *contemporaneous* comparison — it measures whether the market's price agrees with BTC's displacement *at the same instant*. It establishes correlation between two prices, not that BTC leads. It cannot support any trading conclusion, and its prominence on the dashboard invites exactly the misreading this project is supposed to prevent. **P1.**
2. **The same metric is implemented twice with different definitions** — `baseline.py` filters `elapsed_s >= 30`, the dashboard does not. Same name, different numbers. **P1.**
3. **No heartbeat is actually running.** The monitoring code exists in `phantom-research`, but that service is not deployed. Nothing is currently watching the collector. A second 60-hour silent outage is still possible today. **P0 by consequence** — this is the exact failure that already cost 2.5 days of data.

**Clean view logic reviewed line by line:**

**No look-ahead in the cleaning layer:** filters use only window metadata (`open_time`, `close_time`) known at t=0.

## 7. BTC↔Polymarket Calculation — **REPRODUCIBLE BUT MISFRAMED (P1)**

**Definitions actually used:**
- BTC direction = `sign(snapshot.btc_price − resolutions.btc_open)` — window-open price, known at t=0. **No look-ahead.**
- Polymarket direction = `up_mid > 0.5` at the same snapshot.
- Displacement = `(btc_price − btc_open) / btc_open × 10000` bp.
- "Agreement" = the two directions match, measured **at the same timestamp**.

**The critical problem is conceptual, not arithmetic.** This measures whether two prices agree *right now*. A market that simply tracks BTC scores highly by construction. It says nothing about **who moves first**, and therefore nothing tradable. The 90.4% figure is a **sanity check that the market is functioning**, not evidence of an edge.

**Second issue (P1):** the metric exists in two places with different filters —
`app/quant/baseline.py::btc_poly_alignment()` uses `WHERE s.elapsed_s >= 30`;
`dashboard.py::btc_poly_alignment_by_phase()` applies **no** elapsed filter and buckets from 0.
Same concept, two numbers. Must be unified behind one definition.

**Look-ahead check on the wider pipeline:** `clean_markets.winner` is used *only* as an outcome label (convergence, reversal analysis), never as an input feature. Correct as implemented — but this is the single easiest place to introduce a fatal bug later, and deserves a permanent test.

## 8. Outage Detection — **DIRECTIONALLY CORRECT, THRESHOLD TOO TIGHT (P2)**

Recommend classifying gaps by severity (`> 30 min` = OUTAGE, `2–30 min` = DEGRADED) rather than presenting one undifferentiated count that makes 64 h look like 25 equal events.

## 12. Architecture Finding — separation is currently **violated in practice**

**Current:** research SQL (clean views, quality metrics, alignment) lives inside `phantom-collector/dashboard.py`. The independent `phantom-research` engine exists on disk, is fully tested, but is **not in git and not deployed**.

```
CURRENT:  collector ──► Research_DB ──► dashboard.py (collect + analyse + display)
                                         phantom-research (built, idle)

TARGET:   collector ──► Research_DB ──► phantom-research (analyse) ──► research.* ──► dashboard (display only)
```

**Assessment:** acceptable *temporarily* — the dashboard's research queries are read-only projections and caused no harm. But every additional analysis added to `dashboard.py` deepens the coupling and moves computation into a display layer that has no tests. **Do not add further research logic to the dashboard.** Deploy `phantom-research` before the next analysis.

## 15. Challenge to the Proposed Order

Two changes to your sequence, both for the same reason — **measure what the data can actually support, and let experiments shape the tooling rather than the reverse.**

**(a) Do H003 before BTC→Poly latency (you had 8 before 9 — flip them).** Latency is the more exciting question, but our snapshot cadence is ~7 seconds. If Polymarket reacts to BTC in under 7s — which is likely for an automated market — **we physically cannot measure the lag**, and the analysis returns either a null result or an artefact of sampling. H003 by contrast is fully answerable today with large N and clean outcome labels, and it validates a gate already running in production. Do the answerable thing first; treat latency as a *feasibility study* that may conclude "requires 1s collection."

**(b) Run one or two experiments before building the Experiment Lab (you had it at 7).** Designing the schema before knowing what an experiment actually needs produces a lab fitted to imagined workflows. Run H003 manually, note what you wished you had recorded, then build the lab around that. One week later, the schema will be right the first time.

Everything else in your order I agree with — particularly deferring the AI brain until the deterministic layer is stable.
